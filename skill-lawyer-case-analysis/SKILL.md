# lawyer-case-analysis — 案件深度分析

## 触发条件

以下任一情况触发本 skill：
- 用户上传了一份法院裁判文书（判决书/裁定书/调解书）并请求分析
- 用户粘贴了判决书全文
- 用户说"帮我分析这个案子"或类似表达，并附带了文书内容

## 工作流

```
Step 1: 读取用户上传/粘贴的判决书原文
Step 2: 读取 user_profile.json（如存在），了解用户的分析偏好
Step 3: 按分析框架逐维度分析判决书
Step 4: 按输出模板生成9章报告
Step 5: 逐条核对原文，标注每个结论的位置
Step 6: 输出报告
Step 7: 询问反馈："本次分析是否全面？哪些维度需要调整？"
Step 8: 询问保存："要加入策略知识库吗？以后做策略分析时能参考。"
```

## user_profile 适配

如 `../lawyer-strategy/data/user_profile.json` 存在，读取后注意：
- `common_case_types` — 若当前案由是律师常见领域，分析深度自动加深
- `analysis_preferences.focus_areas` — 优先加强这些维度的分析
- `strategy_preferences` — 不影响本 skill（归 strategy）

如文件不存在或路径不可达，正常分析即可。

## 分析框架（11层 → 9章输出）

LLM 内部依次分析以下 11 个维度，但最终输出整合为 9 个章节：

### 维度1: basic_info
提取：案号全称、案由（精准到二级案由）、法院名称（含地域）、审判程序（一审/二审/再审）、判决结果类型（全部支持/部分支持/全部驳回/维持原判/依法改判/调解结案/其他）、双方代理律师姓名及律所

### 维度2: dispute_focus
核心争议：本案双方当事人围绕的核心法律争议是什么
事实摘要：法院认定的基本事实（300字以内）

### 维度3: litigation_motive
导火索事件：什么具体事件引发了本案诉讼
原告诉求（表面）：起诉状中列明的诉讼请求
原告真实动机：表面诉求背后真正的利益诉求（基于判决书推断）
被告立场：被告的抗辩核心观点
本案对律师的参考价值：这个案子的分析对律师有什么启发
风险因子：诉讼中的风险点列表

### 维度4: evidence_battle
逐条分析双方证据：
- 证据名称（标准化，去掉案号/人名/房号等冗余）
- 证据原始描述（保留判决书中的原始表述）
- 举证目的
- 法院采信状态：fully_admitted / partially_admitted / rejected
- 法院认定理由（引用原文或概括）
- 证据权重：high / medium / low
- 证据分类：书证 / 物证 / 电子数据 / 证人证言 / 鉴定意见 / 勘验笔录 / 当事人陈述
- 关键证明点：该证据证明的核心事实（一句话）
举证责任分析：法院关于举证责任分配的论述

### 维度5: legal_paths
逐条分析法律推理路径：
- 路径名称
- 触发该路径的事实（引用原文）
- 引用的法条（必须写标准全称+条号）
- 法条在判决书中的引入位置
- 法院为何在此处引用该法条
- 法院的适用推理过程
- 该路径的结论
- 与其他路径的依赖关系

### 维度6: reasoning_chain
法院从事实认定到法律适用的完整逻辑推演。这不是法律路径的简单罗列，而是展示法院的思维过程：先认定什么事实、再适用什么规则、得出什么中间结论、最终导向什么判决。

### 维度7: turning_points
判决书中有哪些关键事实或事件影响了判决走向。可能是：突现的证据、关键证人的证言、当事人自认、法院对特定事实的自由裁量等。

### 维度8: judge_style
分析审判人员的裁判风格：
- 在证据认定上有何偏好（重书面/重口头/重鉴定）
- 同类案件中的裁判倾向
- 裁判文书的写作风格

### 维度9: patterns
输出标准化、可复用的标签（3-6个字）：
- 事实模式：如"被告缺席""买卖合同未备案""婚前购房"等
- 证据模式：如"微信聊天记录""银行转账记录""欠条原件"等
- 策略模式：如"原告举证不足""被告自认""证据链断裂"等

## 核心约束（5条硬规则）

1. **只基于原文** — 所有分析结论必须基于判决书原文内容，不得编造。不确定的字段明确标注"文中未提及"。
2. **法条引用格式** — 必须写标准法律全称+条号（如"中华人民共和国民法典第一千零七十九条"）。禁止：描述性文字、相对引用（"同上""同解释"）、"无直接依据"类表述。
3. **证据名标准化** — 去掉案号、人名（陈某某/张某等）、具体房号（801房/2栋等）、案涉案号等冗余。例如"801房房产证原件"→"房产证原件"、"彭某手写的《保证书》"→"《保证书》"。详细规则见 references/evidence-rules.md。
4. **不假设心理状态** — 除非判决书明确写了当事人的心理状态描述，否则不得推测当事人的心理活动。
5. **原文可溯** — 每个分析结论标注在判决书原文中的位置（如"判决书第2页'经审理查明'段"）。

## 对话反馈机制

每次分析报告输出完毕后，在末尾附加：

```
---
💬 本次分析你觉得是否全面？有什么维度需要调整？
可以直接告诉我，比如"证据分析再深入一些"或"法律路径更详细点"。
如果满意，可以说"保存到知识库"。
```

## 保存到策略知识库

当用户说"保存"或"保存到知识库"时：
1. 确认路径 `../lawyer-strategy/data/personal.db` 是否存在
2. 存在 → 提取本分析的关键信息写入 personal.db
3. 不存在 → 提示"策略知识库数据库未安装，请先使用 strategy skill"
4. 写入前询问："确认将本案分析加入您的私人策略知识库？仅保存在本机。"
5. 用户拒绝 → 不保存。用户同意 → 写入关键字段：case_no, case_type, court_name, result_type, dispute_focus, analysis（完整分析报告）, created_at

保存成功后告知用户："已保存！以后做策略分析时，这个案例会被参考。"

## 更新 user_profile

分析完成后，更新 `../lawyer-strategy/data/user_profile.json`：
- common_case_types 中对应案由计数 +1
- 如不存在该案由 → 新增 {"type": "XX纠纷", "count": 1}
- total_analyses +1
- last_updated 更新为当前日期
- 如用户给出了反馈（"证据再详细点"等），将其加入 focus_areas

如文件不存在，不报错，正常继续。
